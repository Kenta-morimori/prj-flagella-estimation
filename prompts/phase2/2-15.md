# 2-15: template projection OFF 系を段階的に比較可能化する

## 目的
3 本べん毛・run 条件で `t=100` まで実行しても、論文 Fig.3 の Normal のような後方束化が再現されなかった。  
また、template projection を OFF にすると、単純 OFF では fly-away が起こり、chain-length projection を併用しても即時凝集が見られた。

このことから、現状の template projection は束化阻害だけでなく、flagellum の形状維持や局所配置安定化の一部も肩代わりしている可能性が高い。  
一方で、template projection 自体は論文の力学更新式に明示的には含まれておらず、論文寄りの比較実験を行うには、OFF 系をより自然な拘束で支えられる状態へ段階的に移行する必要がある。

本タスクでは、**template projection OFF 系を比較可能な状態へ段階的に整備する**。  
ただし、変更は **フェーズ単位で明確に分けて実施**し、**ユーザーから次フェーズへの明示的な指示があるまで次に進まないこと**。

---

## 最重要ルール
- **フェーズをまたいで実装しないこと**
- **現在のフェーズで定義された範囲のみ実装すること**
- **各フェーズ完了後、停止して最終報告を行うこと**
- **ユーザーから「次のフェーズへ進んでよい」という明示的な指示があるまで、次フェーズの実装・追加修正・ついでの改善を行わないこと**
- たとえ次フェーズの実装が簡単でも、**勝手に進めないこと**
- 各フェーズで **DoD を満たしたかどうか**を明確に報告すること

---

## 背景整理

### 1. template projection
現状の `src/sim_swim/dynamics/engine.py` では、step 後の拘束処理の中で `_project_flagella_template()` を呼んでいる。  
これは各 flagellum の初期 local helix 形状を保存しておき、現在の基部位置と軸方向に合わせて **べん毛全体をテンプレートへ再配置する**処理である。

この処理は数値安定性には寄与するが、

- helix の横方向形状
- べん毛同士の局所配置
- hook 近傍の向き自由度

まで強く固定しやすく、論文の「hydrodynamics と hook により向きが決まる」構造を弱める可能性がある。

### 2. 論文寄りに形状を支える本筋
論文側では、形状維持は主に以下で支える。

- spring（隣接 bead 間距離）
- bending（局所曲率）
- torsion（局所ねじれ）
- hook
- spring-spring repulsion（非貫通性）

つまり、**形状はテンプレート再投影ではなく、局所ポテンシャルの積み上げで保つ**のが本筋である。

### 3. 現状の問題
- template projection OFF だけでは fly-away する
- chain-length projection 併用でも即時凝集が起こる
- したがって、OFF 系はまだ比較実験可能な状態とは言えない

---

## 今回の全体方針
1PR の中でフェーズを分けて進める。  
ただし、**1フェーズずつ実装し、各フェーズで停止すること**。

フェーズ順:
1. **Phase 1:** template OFF 系の拘束経路整理
2. **Phase 2:** repulsion を比較可能にする
3. **Phase 3:** stiffness scale を config 化する

# Phase 1: template OFF 系の拘束経路整理

## 目的
template projection OFF 系で、何が適用され、何が適用されないかを明確にし、比較実験の前提となる拘束経路を整理する。  
このフェーズでは、**template OFF 時の chain-length projection の扱いを明示的に整えること**が目的であり、repulsion や stiffness の物理量そのものには手を入れない。

## 実施項目
- `template projection ON` 時の既存互換を維持する
- `template projection OFF` 時に `_project_flagella_template()` を呼ばないことを保証する
- `template projection OFF` 時に `_project_flagella_chain_lengths()` を使う経路を明示する
- ただし、chain-length projection の適用方法が強すぎて collapse を助長していないか確認できるよう、実装を読みやすく整理する
- 必要なら、`template projection OFF` 時の chain-length projection の適用位置・回数を見直してよい  
  ただし、**このフェーズでは repulsion パラメータや stiffness scale は変更しないこと**

## このフェーズでやってよいこと
- `engine.py` の拘束分岐整理
- projection 関連 config / 分岐の最小変更
- projection 経路に関する test 追加・修正
- 最小限の可読性改善

## このフェーズでやってはいけないこと
- repulsion パラメータの変更・config 追加
- stiffness scale の変更・config 化
- hook の物理仕様変更
- motor / torque / dt の変更
- 無関係な refactor

## DoD
- [ ] `template projection ON` で既存互換である
- [ ] `template projection OFF` で `_project_flagella_template()` が呼ばれない
- [ ] `template projection OFF` で `_project_flagella_chain_lengths()` を使う経路が明示されている
- [ ] body rigid projection と hook 距離拘束の既存経路を壊していない
- [ ] projection 関連テストが通る
- [ ] このフェーズ完了後に停止し、次フェーズへ進んでいない

## 対応テスト
### 既存テスト
- `tests/test_params.py`
  - template projection の default / override
  - chain-length projection の default / override
- `tests/test_simulation.py`
  - template projection skip の確認
  - chain-length projection 使用経路の確認

### 必要なら追加してよいテスト
- template OFF 時の chain-length projection 適用回数や適用分岐の確認
- hook 距離拘束が OFF 系でも維持されることの確認

## Phase 1 完了時の最終報告で必ず書くこと
1. 変更ファイル一覧
2. 何をどう変更したか
3. template projection ON / OFF 時の拘束経路の違い
4. 実行したテスト
5. DoD の達成状況
6. **次フェーズへは未着手であること**
