## 目的
本タスクの目的は、**菌体の形状維持のために使っている step ごとの `body rigid projection` を、最終的に廃止できる方向へ進めること**である。

現状では、菌体は step ごとに `body rigid projection` により剛体へ戻されている。  
これは論文モデルの「ポテンシャルにより nearly rigid を保つ」という思想と一致しない。

本タスクでは、以下を達成する。

1. `body rigid projection` を **config で ON/OFF 可能**にする
2. **菌体のみ**を対象とした shape stability test を整備する
3. projection OFF 条件で、菌体が **ポテンシャルのみ**で短時間・長時間ともに安定に形状維持できるようにする
4. 3 本べん毛条件で body が崩れる場合に、**CSV から原因候補を説明できる状態**を作る
5. その結果をもって、今後「べん毛 projection がショットノイズ的に効いている可能性」を説明できる土台を作る

## 背景
論文モデルでは、body や flagella の shape は **spring / bending / torsion / repulsion** により *nearly rigid* に保たれる。  
一方、現状実装では body に対して step ごとに `body rigid projection` を適用しており、これは論文モデルからの差分である。

本リポジトリにおける現在の大目的は  
**「3本べん毛が菌体後方で束化し、推進する run 状態を再現すること」**  
である。

ただし、現時点ではべん毛 projection や basal link の扱いが束化阻害要因である可能性が高い。  
その因果を説明するには、まず **菌体側が projection なしでも十分安定**であることを示す必要がある。

そのため本タスクでは、**菌体のみ**を切り出して安定化を行い、次 PR で扱う「べん毛 projection の影響切り分け」の前提を作る。

## 基本方針

### 1. 論文寄りの方針を優先する
まずは、**論文モデルと同型のポテンシャルで shape を維持できるか**を優先して確認すること。

優先対象:
- body spring
- body bending
- 既存の body 構造拘束
- 必要なら構造的な spring 追加

**最初から単に係数を強くするだけの方針は採用しないこと。**

### 2. 係数の強化は最終手段
`stiffness_scales` のような係数増しは、必要性が diagnostics と test で明確になった場合のみ最小限にとどめること。  
最初の解決策として採用しないこと。

### 3. 構造的拘束の追加は許容
菌体の shape maintenance に有効で、かつ論文モデルの「potentials で nearly rigid」に反しにくい対策は採用候補とする。

採用候補:
- 三角柱の各側面長方形に対する **対角 bead 間 spring**
- 断面 shape を保つための最小限の構造的 spring

**実装するかどうかは diagnostics / test を見て判断すること。**  
推測だけで自動採用しないこと。

### 4. 初期条件は自然長・自然角とする
これまでの実装と同様に、**初期条件を自然長・自然角として扱う**こと。  
projection OFF 時に安定性を正しく評価するため、外力 0 の初期状態で内部力が極小になる構成を前提とすること。

## 仕様

## 1. `body rigid projection` を config で ON/OFF 可能にする
現状 `engine.py` の `step()` で適用される `body rigid projection` を、config で切り替え可能にすること。

### 要件
- `SimulationConfig` に body projection 用の config を追加する
- default は **既存互換**
- default は **ON**
- CLI override で OFF にできること
- `with_overrides()` で自然に扱えること

### 命名
以下の命名を使うこと。

```yaml
projection:
  enable_body_rigid_projection: true
````

### 実装上の要件

* `enable_body_rigid_projection=true` のときは現状互換
* `enable_body_rigid_projection=false` のときは `_project_body_rigid()` を呼ばない
* body projection 以外の既存挙動は不用意に壊さないこと

## 2. body-only stability test を追加・整備する

今回の主目的は、**菌体のみ**で projection OFF 時に shape stability が保てるかを検証することである。
したがって、べん毛なし (`n_flagella=0`) の test を整備すること。

### 必須テスト

#### (1) 外力なし静止平衡

条件:

* `n_flagella=0`
* `motor.torque_Nm=0`
* `brownian.enabled=false`
* `projection.enable_body_rigid_projection=false`

確認項目:

* body spring 長誤差が十分小さい
* body bending 角誤差が十分小さい
* NaN / inf が出ない
* COM drift が極小

#### (2) 長時間静止平衡

条件:

* `n_flagella=0`
* `motor.torque_Nm=0`
* `brownian.enabled=false`
* `projection.enable_body_rigid_projection=false`
* `time.duration_s=0.1` 相当以上

確認項目:

* body spring 長誤差が有界
* body bending 角誤差が有界
* triangle area が極端に崩れない
* centerline deviation が極端に増えない
* NaN / inf が出ない

#### (3) 一様外力下の並進

条件:

* `n_flagella=0`
* `projection.enable_body_rigid_projection=false`
* body 全 bead に同方向の外力を加える

確認項目:

* body COM が移動する
* body shape は大きく崩れない
* spring / bending 誤差が許容範囲内

#### (4) 回転 couple 下の回転

条件:

* `n_flagella=0`
* `projection.enable_body_rigid_projection=false`
* 先頭 layer と末尾 layer に反対向きの力を与える

確認項目:

* body が回転する
* body shape は大きく崩れない
* spring / bending 誤差が許容範囲内

#### (5) attach_proxy

条件:

* `n_flagella=0`
* `projection.enable_body_rigid_projection=false`
* 3 本べん毛の attach 点を模した center layer の 3 頂点 bead に、局所集中荷重の proxy を与える

目的:

* 「べん毛がなくても attach 点に局所負荷が入る」状況を擬似的に再現すること
* body-only で局所負荷耐性を測ること
* 3 本べん毛条件の body 崩壊と比較する基準を作ること

確認項目:

* NaN / inf が出ない
* body shape が直ちに壊れない
* どの pair / triplet / face が最初に崩れるかを CSV から追える

### 既存テストの再利用

以下は可能な限り再利用すること。

## 3. 三角柱側面長方形の両対角 spring

これは **実装済み** である。
したがって、今後の修正では以下を前提として扱うこと。

### 固定仕様

* 各 layer 間には 3 つの側面長方形がある
* **各長方形面に対して両対角の spring を持つ**

* 片側対角のみは不可

### 検証

* topology test で side-brace spring 本数を確認すること
* 追加修正で本数や意味を壊さないこと

### 3. 典型的な解釈

* **外力 0 で落ちる**

* **一様並進だけ落ちる**

* **attach_proxy だけ落ちる**
  attach 点近傍の局所荷重耐性不足を疑う
* **3 本べん毛条件だけ落ちる**
  motor / hook / flag projection 由来の負荷を疑う

* diagnostics 根拠なしに global stiffness を増やさない
* hidden projection / hidden correction を再導入しない
* test 閾値だけを緩めない
* notes 更新を省略しない
