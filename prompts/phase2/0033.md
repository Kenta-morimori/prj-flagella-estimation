## 背景

直前の PR #32 / Issue #29 では，body-only 条件を切り出して安定性を確認した。

- `projection.enable_body_rigid_projection` を追加し，body rigid projection を ON/OFF 可能にした
- body 側面長方形に **両対角 spring** を追加した
- `n_flagella=0` では，`projection.enable_body_rigid_projection=false` でも body は安定した
- 一方で `n_flagella=3` では body 崩壊が起きうるため，主因は body 単体ではなく **hook / flagella が body に返す負荷**の可能性が高い

ただし，ここで注意すべき点がある。  
`n_flagella=0` の body-only 安定は，**body が自発的に崩れにくい**ことは示すが，**motor ON 時に flagellum から返る実負荷に対して十分に安定**であることまでは保証しない。  
したがって，body-only 安定は必要な切り分けではあるが，**motor equivalent load に対する十分条件ではない**。

このため，次の切り分け対象として **hook / flagella projection 側** を扱った。

### 追加で今回分かったこと
PR #34 の Stage 1 (`n_flagella=1`, `motor.torque_Nm=0`) の 2x2 比較では，

- hook OFF / flagella OFF
- hook OFF / flagella ON
- hook ON / flagella OFF
- hook ON / flagella ON

の **全条件で形状崩壊**が生じた。

その後，`paper_table1`・`n_flagella=1`・`motor.torque_Nm=0`・all projection OFF で再検証したところ，

- 初期 bend / torsion / hook angle は target と実質一致
- `paper_table1` と all projection OFF は正しく反映
- `fd_eps_over_b=0.1` では step 0 から崩壊
- `fd_eps_over_b=1e-3` および `1e-4` では，菌体・べん毛とも静止平衡をほぼ維持

であった。

したがって，projection の有無そのものや，paper mode における初期 shape mismatch は主因ではない可能性が高い。  
Phase A の結論として，**torsion force 実装側，特に有限差分幅 `fd_eps_m` が大きすぎることが崩壊原因であり，current candidate は `fd_eps_over_b=1e-3`** とする。

一方で，`paper_table1`・`n_flagella=1`・all projection OFF・`fd_eps_over_b=1e-3` でも，

- `motor.torque_Nm = 4e-18`
- `motor.torque_Nm = 1e-18`
- `motor.torque_Nm = 5e-20`

のいずれでも `motor ON` では基部から崩壊した。  
このため，次の主因候補は **`compute_motor_forces()` による torque→force couple 分配**，または **motor 印加時の basal 幾何拘束不足** であると考えていた。

その後，Phase D / E と追加比較により，以下が分かった。

- `hook_length` により hook 長の崩れ自体は抑えられる
- `hook_length + basal_link_direction` でも flagella chain 長の崩れは抑えられない
- `chain_length` のみ ON では短時間安定
- `template` のみ ON でも短時間安定
- `hook_basal_ref` は即時破綻
- `Ta + Tb = Ttot` の split residual は機械誤差レベルであり，`compute_motor_forces()` の gross bug は主因ではなさそう

したがって，現時点では

- **projection は数値安定化には有効**
- しかし **論文の potential 主体モデルより強い人工拘束**
- 現実装では `motor ON` 時の flagella shape-maintenance が不足しており，`chain_length` または `template` がその不足を人工的に埋めている可能性が高い

と整理する。

さらに body-only の motor equivalent load 比較により，

- baseline / `attach_proxy` では body はほぼ無変形
- 旧 `pure_couple` では body の局所ねじれが発生
- 作用点・分配を見直した `pure_couple_distributed` では body-only short-time 条件で nearly rigid に近い応答へ改善

した。  
したがって，旧 pure couple の問題は **body 一般の不安定性よりも load distribution 起因**の可能性が高く，body-only proxy は `pure_couple_distributed` を採用する。

また、projection を比較にすら使わず原因切り分けしたいという方針に基づき，次段では

1. **菌体のみ**
2. **菌体 + minimal basal stub**
3. **菌体 + full flagella**

の3段階を **すべて projection OFF** で比較し，どの段階から破綻が現れるかを評価する。

ここで **minimal basal stub** とは，
- body attach bead
- first bead
- second bead  
までを残し，
- その先の helix chain を持たない最小基部系

を指す。  
これは「菌体 + フックのみ」を実装上明確に定義した比較系である。

よって，この prompt では

1. projection の個別 ON/OFF 化
2. **新規比較・新規評価での projection default OFF 化**
3. **論文準拠の一意な初期べん毛生成**
4. **初期状態の幾何情報の明示的 logging / summary 出力**
5. **torsion force 実装（とくに `fd_eps_m`）の検証**
6. **motor diagnostics の追加と motor/basal 問題の切り分け**
7. **projection を不採用にした上で，body-only に motor equivalent load を与えた形状安定性検証**
8. **body-only / minimal basal stub / full flagella の3段階比較系整備**
9. **flagella 側局所破綻モードを特定するための diagnostics 追加実装**
10. **`flag_torsion_stiffness_scale` / `flag_bend_stiffness_scale` の扱いに関する結論整理**

を扱う。

## 現 `main` / 現ブランチの把握事項（前提）

実装前に必ず現コードを読んで確認すること。

### 1. 現在 config 化されている projection は body 側のみ
- `projection.enable_body_rigid_projection` は存在する
- hook / flagella projection を個別に切る config はまだない

### 2. `step()` の最後で hook / flagella 側 projection が常時走る
現 `main` では，body rigid projection の後に hook / flagella 側 projection が必ず適用される。

### 3. hook / flagella projection の中身は少なくとも以下に分かれている
- hook length の補正
- basal link direction の補正
- flagella chain length の補正
- flagella template の補正

### 4. hook の現在の物理定義
- `attach bead - first flag bead` は spring pair に入っている
- hook は `(attach, first, second)` の triplet で表されている
- motor triplet も同じ triplet を使っている

### 5. 初期化時の幾何
- `attach -> first` は body layer centroid から見た radial 方向に短く置かれている
- `first -> second` の基底接線は rear direction に概ね揃えられている
- これは **初期条件** であり，時間発展中もその方向を hard に固定すべきとは限らない

### 6. 現状の問題点
- 旧来の `legacy_radius_pitch` では，初期べん毛形状と target bend / torsion が独立であり，初期 mismatch を生みうる
- 一方，`paper_table1` では初期 shape mismatch は大きく改善された
- Phase A により，`fd_eps_m` が崩壊主因だったことが切れた
- current candidate は **`fd_eps_over_b=1e-3`**
- しかし `motor ON, n_flagella=1` では，`4e-18`, `1e-18`, `5e-20` のいずれでも崩壊した
- Phase D / E と追加比較により，`hook_basal` だけでは不足し，`chain_length` または `template` が入ると短時間安定することが分かった
- body-only proxy 比較により，旧 `pure_couple` は不適切な load distribution であり，`pure_couple_distributed` が妥当な proxy と分かった
- ただし現状は，**べん毛のどの局所位置・どの mode が最初に破綻するか**までは特定できていない
- したがって，現時点の主因候補は **gross な motor split bug** ではなく，**motor ON 時に potential のみで flagella shape を保てないこと** にある
- engine 内には `flag_bend_stiffness_scale = 300`，`flag_torsion_stiffness_scale = 300`，`body_stiffness_scale = 50` があり，これらは論文の一次入力ではなく repo 独自の補助係数である可能性が高い
- ただし，現時点では **body 一般の剛性不足を先に疑う段階ではなく，minimal basal stub / full flagella 比較と flagella 局所破綻モードの特定を優先**すべきである

## 今回の設計方針（重要）

### 方針 1: hook は「角度固定」ではなく「距離中心」で扱う
論文モデル寄りに，hook については以下を採ること。

- **常時の角度固定は行わない**
- `attach-first` の距離は **projection ではなく spring** を主拘束とする
- hook angle は，body 交差を防ぐための **one-sided な hook 制約**として扱うのは可
- `basal link direction` の hard projection は，論文より強い拘束の可能性があるため，個別に ON/OFF 可能にする

### 方針 2: hook と flagella の projection はまとめてではなく分解する
最低限，以下を個別に ON/OFF 可能にすること。

- `projection.enable_hook_length_projection`
- `projection.enable_basal_link_direction_projection`
- `projection.enable_flagella_chain_length_projection`
- `projection.enable_flagella_template_projection`

### 方針 3: projection の default を OFF にする
- **新規比較・新規評価では projection default を OFF とする**
- 既存互換 run が必要な場合のみ，明示的に ON を指定する
- 既存テスト・既存出力に影響する場合は，互換性維持の扱いを `run.log` / notes に明記する

### 方針 4: projection は診断用に限定する
- `chain_length` と `template` は **数値安定化には有効**
- しかし **論文モデルの potential 主体実装より強い人工拘束**
- したがって，**これらを最終解として採用しないこと**
- 役割は **どの破綻 mode を抑えているかを診断するための比較用拘束** に限定すること
- **projection あり条件を安定性評価の主指標・相対比較の基準にしないこと**
- **相対評価の中心は all projection OFF とすること**

### 方針 5: 比較は projection ではなく系の構成で段階化する
projection 比較ではなく，以下の3段階比較を主軸にする。

1. **body-only**
2. **body + minimal basal stub**
3. **body + full flagella**

ここで、
- body-only は `n_flagella=0` と body-equivalent-load proxy を使う
- minimal basal stub は attach / first / second のみを残す最小基部系
- full flagella は通常の `n_flagella=1` paper_table1 系

とする。

### 方針 6: 初期べん毛は「論文の離散モデル量」を正とする
初期べん毛形状は，`helix_init.radius_over_b` と `helix_init.pitch_over_b` を一次入力とするのではなく，**論文 Table 1 に対応する離散モデル量**を source of truth とすること。

少なくとも以下を一次入力候補として扱う。

- `bond_L_over_b`
- `potentials.bend.theta0_deg.normal`
- `potentials.torsion.phi0_deg.normal`
- `n_beads_per_flagellum`
- `n_flagella`

`radius_over_b` と `pitch_over_b` は，必要なら **派生値として計算・記録**する。  
paper 準拠 mode では，`radius/pitch` を手入力で同時指定して source of truth を二重化しないこと。

### 方針 7: 初期状態の情報を log / summary に明示する
実長さの解釈や bead 数の妥当性を後から確認できるよう，初期状態について以下を必ず記録すること。

- 各べん毛の bead 数
- 各べん毛の contour length / end-to-end length
- `attach-first` 長さ
- flagella bond 長さ統計
- 初期 hook angle
- 初期 bend / torsion の実測値と target 値との差
- 派生された helix radius / pitch
- 初期 tangent と rear direction の角度
- 使用した init mode と source-of-truth 項目

### 方針 8: torsion force の問題はフェーズを分けて扱う
このタスクでは，以下の順で扱うこと。

#### Phase A（完了済み）
- **`fd_eps_m` を 100〜1000 倍小さくする sweep** を行った
- `paper_table1`・`n_flagella=1`・`motor.torque_Nm=0`・all projection OFF で再検証した
- `fd_eps_over_b=0.1` では崩壊
- `fd_eps_over_b=1e-3` および `1e-4` では静止平衡を維持
- current candidate は **`fd_eps_over_b=1e-3`**

#### Phase B（完了済み）
- `paper_table1`・`n_flagella=1`・`motor.torque_Nm>0`・all projection OFF・`fd_eps_over_b=1e-3` を検証した
- `motor.torque_Nm = 4e-18`, `1e-18`, `5e-20` のいずれでも崩壊した
- したがって，**torsion finite-difference 問題は切れたが，motor ON の不安定性は未解決**である

#### Phase C（完了済み）
- **motor diagnostics を追加**し，step 0 / 1 の basal triplet まわりを可視化できるようにした
- `paper_table1`・`n_flagella=1`・`motor.torque_Nm=5e-20`・`fd_eps_over_b=1e-3` を基準条件とした
- 少なくとも以下を出力できるようにした
  - `motor_ra_len_um`
  - `motor_rb_len_um`
  - `motor_Ta_norm`
  - `motor_Tb_norm`
  - `motor_Fa_norm`
  - `motor_Fb_norm`
  - `motor_axis_vs_rear_direction_angle_deg`
  - `motor_attach_force_norm`
  - `motor_first_force_norm`
  - `motor_second_force_norm`
  - `motor_degenerate_axis_count`
  - `motor_split_rank_deficient_count`
  - `motor_bond_length_clipped_count`

#### Phase D（実施結果: 2026-03-31）
- 比較条件は `paper_table1`・`n_flagella=1`・`motor.torque_Nm=5e-20`・`fd_eps_over_b=1e-3`
- 4ケース比較:
  - all OFF
  - `hook_length` のみ ON
  - `hook_length + basal_link_direction` ON
  - all ON
- hook 長は `hook_length` で維持されるが，flagella chain は維持されない
- `hook_basal` は安定化の十分条件ではなかった

#### Phase E（実施結果: 2026-03-31）
- split residual は機械誤差レベル
- second bead 方向の過大分配は緩和
- それでも `hook_basal` は局所伸長崩壊を止められない
- 結論として，主因は split 式の gross bug より **shape-maintenance 欠如** 側が有力

#### Phase F（body-only proxy 整備）
- baseline / `attach_proxy` では body はほぼ無変形
- old `pure_couple` では局所ねじれ
- `pure_couple_distributed` では short-time body-only 条件で nearly rigid に近い応答へ改善
- body-only proxy は `pure_couple_distributed` を採用

#### Phase G（今回の次アクション）
- projection ではなく系構成比較を主軸にする
- **body-only / minimal basal stub / full flagella** の3段階を all projection OFF で比較する
- 次段では **minimal basal stub が壊れるかどうか** を確認する
- その上で full flagella で初めて壊れるなら，原因は helix chain 側に強く絞れる
- 逆に minimal basal stub で既に壊れるなら，basal/hook 近傍のモデル不足が有力になる

### 方針 9: 追加 scale は論文値ではない前提で扱う
- `flag_bend_stiffness_scale`
- `flag_torsion_stiffness_scale`
- `body_stiffness_scale`

は論文の一次入力そのものではなく，repo 独自の補助係数とみなすこと。  
したがって，paper mode ではその使用可否・既定値を慎重に扱うこと。

## D. 論文準拠の初期べん毛生成 mode を追加する

### 目的
初期べん毛形状と Normal 状態の平衡 bend / torsion が一致するようにし，`motor.torque_Nm=0` でも初期緩和反力で崩れない方向へ寄せる。

### 必須方針
- 既存の `helix_init.radius_over_b` / `pitch_over_b` ベース初期化は **legacy mode** として残してよい
- ただし，新たに **paper 準拠 mode** を追加すること
- `conf/sim_swim.yaml` の既定は，可能なら paper 準拠 mode に寄せること
- paper 準拠 mode では，初期べん毛形状を **target の `bond_L_over_b`, `theta0_deg.normal`, `phi0_deg.normal`, bead 数** に整合するよう生成すること

### config 例
- `flagella.init_mode: "paper_table1" | "legacy_radius_pitch"`
- `flagella.n_beads_per_flagellum`
- `flagella.bond_L_over_b`
- `potentials.bend.theta0_deg.normal`
- `potentials.torsion.phi0_deg.normal`
- `flagella.stub_mode: none | minimal_basal_stub | full_flagella`

### 制約
- paper 準拠 mode では，`radius_over_b` / `pitch_over_b` を source of truth にしない
- 必要なら `radius/pitch` は builder 内で導出し，summary に出す
- source of truth が二重化する入力は避ける
- 初期化のために hidden correction を step 側へ持ち込まないこと
