## 背景

直前の PR #32 / Issue #29 では，body-only 条件を切り出して安定性を確認した。

- `projection.enable_body_rigid_projection` を追加し，body rigid projection を ON/OFF 可能にした
- body 側面長方形に **両対角 spring** を追加した
- `n_flagella=0` では，`projection.enable_body_rigid_projection=false` でも body は安定した
- 一方で `n_flagella=3` では body 崩壊が起きうるため，主因は body 単体ではなく **hook / flagella が body に返す負荷**の可能性が高い

このため，次の切り分け対象として **hook / flagella projection 側** を扱った。

### 追加で今回分かったこと
PR #34 の Stage 1 (`n_flagella=1`, `motor.torque_Nm=0`) の 2x2 比較では，

- hook OFF / flagella OFF
- hook OFF / flagella ON
- hook ON / flagella OFF
- hook ON / flagella ON

の **全条件で形状崩壊**が生じた。

その後，`paper_table1`・`n_flagella=1`・`motor.torque_Nm=0`・all projection OFF で再検証したところ，

- 初期 bend / torsion / hook angle は target と実質一致
- `paper_table1` と all projection OFF は正しく反映
- `fd_eps_over_b=0.1` では step 0 から崩壊
- `fd_eps_over_b=1e-3` および `1e-4` では，菌体・べん毛とも静止平衡をほぼ維持

であった。

したがって，projection の有無そのものや，paper mode における初期 shape mismatch は主因ではない可能性が高い。  
Phase A の結論として，**torsion force 実装側，特に有限差分幅 `fd_eps_m` が大きすぎることが崩壊原因であり，current candidate は `fd_eps_over_b=1e-3`** とする。

一方で，`paper_table1`・`n_flagella=1`・all projection OFF・`fd_eps_over_b=1e-3` でも，

- `motor.torque_Nm = 4e-18`
- `motor.torque_Nm = 1e-18`
- `motor.torque_Nm = 5e-20`

のいずれでも `motor ON` では基部から崩壊した。  
このため，次の主因候補は **`compute_motor_forces()` による torque→force couple 分配**，または **motor 印加時の basal 幾何拘束不足** である。

よって，この prompt では

1. projection の個別 ON/OFF 化
2. **論文準拠の一意な初期べん毛生成**
3. **初期状態の幾何情報の明示的 logging / summary 出力**
4. **torsion force 実装（とくに `fd_eps_m`）の検証**
5. **motor diagnostics の追加と motor/basal 問題の切り分け**
6. **`flag_torsion_stiffness_scale` / `flag_bend_stiffness_scale` の扱いに関する結論整理**

を扱う。

## 現 `main` / 現ブランチの把握事項（前提）

実装前に必ず現コードを読んで確認すること。

### 1. 現在 config 化されている projection は body 側のみ
- `projection.enable_body_rigid_projection` は存在する
- hook / flagella projection を個別に切る config はまだない

### 2. `step()` の最後で hook / flagella 側 projection が常時走る
現 `main` では，body rigid projection の後に hook / flagella 側 projection が必ず適用される。

### 3. hook / flagella projection の中身は少なくとも以下に分かれている
- hook length の補正
- basal link direction の補正
- flagella chain length の補正
- flagella template の補正

### 4. hook の現在の物理定義
- `attach bead - first flag bead` は spring pair に入っている
- hook は `(attach, first, second)` の triplet で表されている
- motor triplet も同じ triplet を使っている

### 5. 初期化時の幾何
- `attach -> first` は body layer centroid から見た radial 方向に短く置かれている
- `first -> second` の基底接線は rear direction に概ね揃えられている
- これは **初期条件** であり，時間発展中もその方向を hard に固定すべきとは限らない

### 6. 現状の問題点
- 旧来の `legacy_radius_pitch` では，初期べん毛形状と target bend / torsion が独立であり，初期 mismatch を生みうる
- 一方，`paper_table1` では初期 shape mismatch は大きく改善された
- Phase A により，`fd_eps_m` が崩壊主因だったことが切れた
- current candidate は **`fd_eps_over_b=1e-3`**
- しかし `motor ON, n_flagella=1` では，`4e-18`, `1e-18`, `5e-20` のいずれでも崩壊した
- 現在の主因候補は，**motor torque を basal triplet へ force couple として分配する実装**，または **motor 印加時の basal 幾何拘束不足**
- engine 内には `flag_bend_stiffness_scale = 300`，`flag_torsion_stiffness_scale = 300`，`body_stiffness_scale = 50` があり，これらは論文の一次入力ではなく repo 独自の補助係数である可能性が高い
- ただし，現時点では **scale 変更を先に行うのではなく，まず motor diagnostics と basal 最小比較を先行**すべきである

## 今回の設計方針（重要）

### 方針 1: hook は「角度固定」ではなく「距離中心」で扱う
論文モデル寄りに，hook については以下を採ること。

- **常時の角度固定は行わない**
- `attach-first` の距離は **projection ではなく spring** を主拘束とする
- hook angle は，body 交差を防ぐための **one-sided な hook 制約**として扱うのは可
- `basal link direction` の hard projection は，論文より強い拘束の可能性があるため，個別に ON/OFF 可能にする

### 方針 2: hook と flagella の projection はまとめてではなく分解する
最低限，以下を個別に ON/OFF 可能にすること。

- `projection.enable_hook_length_projection`
- `projection.enable_basal_link_direction_projection`
- `projection.enable_flagella_chain_length_projection`
- `projection.enable_flagella_template_projection`

既定値は **すべて `true`** とし，現状互換を保つこと。

### 方針 3: 切り分けは段階的に行う
`hook OFF / flagella ON` だけで因果を断定しないこと。  
少なくとも以下の比較ができるようにする。

1. hook ON / flagella ON
2. hook OFF / flagella ON
3. hook ON / flagella OFF
4. hook OFF / flagella OFF

ここで
- hook 系 = `hook length` + `basal link direction`
- flagella 系 = `chain length` + `template`

とする。

### 方針 4: 初期べん毛は「論文の離散モデル量」を正とする
初期べん毛形状は，`helix_init.radius_over_b` と `helix_init.pitch_over_b` を一次入力とするのではなく，**論文 Table 1 に対応する離散モデル量**を source of truth とすること。

少なくとも以下を一次入力候補として扱う。

- `bond_L_over_b`
- `potentials.bend.theta0_deg.normal`
- `potentials.torsion.phi0_deg.normal`
- `n_beads_per_flagellum`
- `n_flagella`

`radius_over_b` と `pitch_over_b` は，必要なら **派生値として計算・記録**する。  
paper 準拠 mode では，`radius/pitch` を手入力で同時指定して source of truth を二重化しないこと。

### 方針 5: 初期状態の情報を log / summary に明示する
実長さの解釈や bead 数の妥当性を後から確認できるよう，初期状態について以下を必ず記録すること。

- 各べん毛の bead 数
- 各べん毛の contour length / end-to-end length
- `attach-first` 長さ
- flagella bond 長さ統計
- 初期 hook angle
- 初期 bend / torsion の実測値と target 値との差
- 派生された helix radius / pitch
- 初期 tangent と rear direction の角度
- 使用した init mode と source-of-truth 項目

### 方針 6: torsion force の問題はフェーズを分けて扱う
このタスクでは，以下の順で扱うこと。

#### Phase A（完了済み）
- **`fd_eps_m` を 100〜1000 倍小さくする sweep** を行った
- `paper_table1`・`n_flagella=1`・`motor.torque_Nm=0`・all projection OFF で再検証した
- `fd_eps_over_b=0.1` では崩壊
- `fd_eps_over_b=1e-3` および `1e-4` では静止平衡を維持
- current candidate は **`fd_eps_over_b=1e-3`**

#### Phase B（完了済み）
- `paper_table1`・`n_flagella=1`・`motor.torque_Nm>0`・all projection OFF・`fd_eps_over_b=1e-3` を検証した
- `motor.torque_Nm = 4e-18`, `1e-18`, `5e-20` のいずれでも崩壊した
- したがって，**torsion finite-difference 問題は切れたが，motor ON の不安定性は未解決**である

#### Phase C（次の必須検証）
- **motor diagnostics を追加**し，step 0 / 1 の basal triplet まわりを可視化する
- `paper_table1`・`n_flagella=1`・`motor.torque_Nm=5e-20`・`fd_eps_over_b=1e-3` を基準条件とする
- 少なくとも以下を出力すること
  - `motor_ra_len_um` (`attach-first`)
  - `motor_rb_len_um` (`first-second`)
  - `motor_Ta_norm`
  - `motor_Tb_norm`
  - `motor_Fa_norm`
  - `motor_Fb_norm`
  - `motor_axis_vs_rear_direction_angle_deg`
  - `motor_attach_force_norm`
  - `motor_first_force_norm`
  - `motor_second_force_norm`
  - 必要なら `degenerate_axis_count`, `split_rank_deficient_count`, `bond_length_clipped_count`
- 不明点や出力定義が曖昧な場合は，ここで事前に質問すること

#### Phase D（次の比較実験）
- `motor.torque_Nm=5e-20` を固定し，以下を比較する
  - all projection OFF
  - `hook_length` のみ ON
  - `hook_length + basal_link_direction` ON
  - all projection ON
- 目的は，
  - **motor force 分配自体が悪いのか**
  - **motor 印加時に basal 幾何拘束が不足しているのか**
  を切ること

#### Phase D（実施結果: 2026-03-31）
- 比較条件は `paper_table1`・`n_flagella=1`・`motor.torque_Nm=5e-20`・`fd_eps_over_b=1e-3`
- 出力先:
  - `outputs/inspect_motor_diag/phaseD/all_off/.../sim/step_summary.csv`
  - `outputs/inspect_motor_diag/phaseD/hook_only/.../sim/step_summary.csv`
  - `outputs/inspect_motor_diag/phaseD/hook_basal/.../sim/step_summary.csv`
  - `outputs/inspect_motor_diag/phaseD/all_on/.../sim/step_summary.csv`
- 主要観測（last step）:
  - all OFF: `hook_len_mean_over_b=37.98`, `flag_bond_len_mean_over_b=92.81`
  - hook only: `hook_len_mean_over_b=0.25`, `flag_bond_len_mean_over_b=82.81`
  - hook + basal: `hook_len_mean_over_b=0.25`, `flag_bond_len_mean_over_b=94.69`
  - all ON: `hook_len_mean_over_b=0.25`, `flag_bond_len_mean_over_b=0.58`
- 解釈:
  - `hook_length` を戻すと hook 長は維持されるが，flagella chain は維持されない
  - `hook_length + basal_link_direction` でも flagella chain 崩壊は止まらない
  - 以前の仮説「`hook_length + basal_link_direction` があれば十分」は，Phase D の比較結果では支持されなかった
  - 少なくとも `torque=5e-20`, `n_flagella=1`, `fd_eps_over_b=1e-3` 条件では，flagella chain / template 側の拘束がないと幾何破綻を防げなかった
  - 少なくともこの比較では，**「basal 最小拘束だけで安定化」には至らず，motor ON 不安定性は basal のみでは説明しきれない**
  - 次段では，`compute_motor_forces()` の torque→force couple 分配（Phase E）を優先検証する

#### Phase E（条件付き）
- Phase C / D を行っても，最小拘束を戻してなお崩れる場合に限り，
  - `compute_motor_forces()` の torque 分配法
  - `f = (T × r) / ||r||^2` の適用位置
  - body / first / second bead への force couple の入れ方
  を見直してよい
- Phase E では，少なくとも次の3点を優先検証対象として明示すること
  - `Ta + Tb = Ttot` の split が妥当か
  - `f = (T × r) / ||r||^2` による force couple が短い basal link で過大になっていないか
  - attach / first / second bead への分配が局所伸長を直接作っていないか
- その後にのみ，
  - `flag_torsion_stiffness_scale`
  - `flag_bend_stiffness_scale`
  の扱いを検討してよい

#### Phase E（実施結果: 2026-03-31）
- 比較条件は `paper_table1`・`n_flagella=1`・`motor.torque_Nm=5e-20`・`fd_eps_over_b=1e-3`
- 出力先:
  - `outputs/inspect_motor_diag/phaseE/all_off/2026-03-31/205653/sim/step_summary.csv`
  - `outputs/inspect_motor_diag/phaseE/hook_basal/2026-03-31/205711/sim/step_summary.csv`
  - `outputs/inspect_motor_diag/phaseE/all_on/2026-03-31/205730/sim/step_summary.csv`
- `Ta + Tb = Ttot` 妥当性（split residual）:
  - all OFF: `motor_split_residual_norm max=5.98e-29`
  - hook + basal: `motor_split_residual_norm max=1.14e-26`
  - all ON: `motor_split_residual_norm max=1.09e-28`
  - いずれも機械誤差レベルで，`Ta + Tb = Ttot` の split 自体は破綻していない
- 短い basal link での force 過大化チェック:
  - `motor_attach_force_norm` / `motor_first_force_norm` は hook+basal / all ON で概ね `~2.0e-13`（ピーク）
  - 一方で `motor_second_force_norm` は all ON で `~1e-22` とほぼゼロ，hook+basal でも `~1e-15` 程度に抑制
  - force-aware split により third bead 方向の不要な分配は抑えられたが，attach/first 近傍の高荷重そのものは残る
- 局所伸長への影響:
  - hook + basal: `flag_bond_rel_err_max last=3.62e+02`（依然として大崩壊）
  - all ON: `flag_bond_rel_err_max last=5.48e-16`（実質ゼロ）
  - よって，motor split 改良のみでは `hook + basal` 条件の幾何崩壊を止められず，chain/template 側拘束の有無が支配的
- 結論:
  - Phase E の3観点のうち，(1) split の整合性は OK，(2) third bead への過大分配は緩和，(3) それでも `hook + basal` では局所伸長崩壊が継続
  - 本件の主因は「split式の単純な誤り」よりも，motor ON 時に chain/template 拘束なしで形状保持できないモデル構造側にある可能性が高い

### 方針 7: 追加 scale は論文値ではない前提で扱う
- `flag_bend_stiffness_scale`
- `flag_torsion_stiffness_scale`
- `body_stiffness_scale`

は論文の一次入力そのものではなく，repo 独自の補助係数とみなすこと。  
したがって，paper mode ではその使用可否・既定値を慎重に扱うこと。

## D. 論文準拠の初期べん毛生成 mode を追加する

### 目的
初期べん毛形状と Normal 状態の平衡 bend / torsion が一致するようにし，`motor.torque_Nm=0` でも初期緩和反力で崩れない方向へ寄せる。

### 必須方針
- 既存の `helix_init.radius_over_b` / `pitch_over_b` ベース初期化は **legacy mode** として残してよい
- ただし，新たに **paper 準拠 mode** を追加すること
- `conf/sim_swim.yaml` の既定は，可能なら paper 準拠 mode に寄せること
- paper 準拠 mode では，初期べん毛形状を **target の `bond_L_over_b`, `theta0_deg.normal`, `phi0_deg.normal`, bead 数** に整合するよう生成すること

### config 例
命名は適宜調整してよいが，意図は以下を満たすこと。

- `flagella.init_mode: "paper_table1" | "legacy_radius_pitch"`
- `flagella.n_beads_per_flagellum`
- `flagella.bond_L_over_b`
- `potentials.bend.theta0_deg.normal`
- `potentials.torsion.phi0_deg.normal`

### 制約
- paper 準拠 mode では，`radius_over_b` / `pitch_over_b` を source of truth にしない
- 必要なら `radius/pitch` は builder 内で導出し，summary に出す
- source of truth が二重化する入力は避ける
- 初期化のために hidden correction を step 側へ持ち込まないこと

## 定量基準（hard gate と soft gate を分ける）

## Hard gate（根拠が比較的あるもの）
既存テスト・既存実装に寄せて，まず以下を commit 前基準候補とする。

### hard gate の適用範囲
hard gate の閾値は，まずは **`n_flagella=1` かつ `motor.torque_Nm=0` の追加テストにのみ適用**する。  
`motor.torque_Nm>0` および `n_flagella=3` は，現段階では比較可能性と diagnostics による原因説明を重視する **soft gate** として扱う。

### body
body-only / 低負荷条件では既存基準を維持すること。  
既存閾値と整合しない変更を入れる場合は理由を明記すること。

### hook length
`attach-first` の距離について，少なくとも 1 flagellum / torque=0 / 短時間条件では以下を目安にする。

- mean: rest length に対して **±5% 以内**
- min/max: rest length に対して **0.85〜1.15 倍** 以内

### flagella bond length
1 flagellum / torque=0 / 短時間条件で以下を目安にする。

- mean: rest length に対して **±2% 以内**
- min/max: rest length に対して **0.95〜1.05 倍** 以内

### initial-geometry hard gate
paper 準拠 mode では，少なくとも `n_flagella=1` の初期状態で，
- target と measured の bond 長さ誤差
- target と measured の bend 角誤差
- target と measured の torsion 角誤差

が十分小さいことを確認する。

### torsion-implementation hard gate
少なくとも `paper_table1`・`n_flagella=1`・`motor.torque_Nm=0`・all projection OFF の短時間条件で，
- `fd_eps_over_b=1e-3` が静止平衡を保つこと
- `fd_eps_over_b=0.1` との差が説明できること

を確認する。

### bend / torsion
既存テストにあるなら，1 flagellum 条件でその閾値を流用してよい。
