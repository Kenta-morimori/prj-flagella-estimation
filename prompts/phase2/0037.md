## 本ドキュメントの目的
本タスクの第一目的は、力学演算を以下の3モードで明示的に扱えるようにし、
演算中に菌体・べん毛が弾ける挙動を許容しない検証枠組みを整えることである。

- body_only
- body_hook
- body_hook_flagella

本フェーズでは、固定の絶対閾値を先に決めることはしない。
代わりに、破綻を取りこぼさない厳格テスト（不変条件・一貫性・悪化検知）を優先する。

---

## DoD

### PhaseA DoD（必須）
1. 1本べん毛・トルクなしで短時間安定化できること
- 想定構成: n_flagella=1, stub_mode=minimal_basal_stub, motor.torque_Nm=0
- short run の全stepで以下を満たす:
  - pos_all_finite=True
  - any_nan=False
  - any_inf=False
- local diagnostics 列が欠損せず、有限値として継続出力されること

2. 構造切替が正しく反映されること
- body_only / body_hook / body_hook_flagella の各モードで、構築される構造（bead/spring/triplet/quad）が意図どおりであること

### PhaseB DoD（必須: 要件整理、可能なら着手）
1. トルクあり安定化に必要な要件を明文化すること
- 破綻防止観点で最低限以下を整理する:
  - motor split の特異/退化検知
  - basal近傍リンクの急伸長検知
  - full chain 側の崩れ検知
  - first-fail の定義

2. 可能なら実装着手し、短時間で検証可能にすること
- 想定構成:
  - PhaseB-minimal: n_flagella=1, stub_mode=minimal_basal_stub, motor.torque_Nm!=0
  - PhaseB-full: n_flagella=1, stub_mode=full_flagella, motor.torque_Nm!=0
- 固定閾値なしでも fail 判定できるテスト（悪化傾向・欠損・非有限）を実装すること

3. full_flagella + motor off の静止実行を厳格にクリアできること
- 判定は `pos_all_finite=True` だけでは不十分で、形状崩壊を取りこぼさないこと
- 必須条件:
  - `hook_count == 1` と `flag_intra_count == 10` が全 step で維持されること
  - `hook_len_mean_over_b`, `local_attach_first_rel_err`, `flag_bond_rel_err_max`, `local_first_torsion_err_deg` が late-time で runaway growth しないこと
- クリア条件は step_summary.csv の時系列で判定し、見た目の印象では決めないこと

## DoD以外に必須で定義する項目

### 1. Non-goals（今回やらないこと）
- n_flagella > 1 の安定bundle再現
- run/tumble の本格再開
- wall effect 導入
- 論文Fig.3レベルの最終再現 hard gate 化
- projection系拘束の再導入
- flagellum全長 7.5b の厳密再設計

### 2. Scope boundary（今回の境界）
- 必須達成: PhaseA DoD
- 必須達成: PhaseB 要件整理
- 可能なら実施: PhaseB の短時間安定化着手

### 3. Acceptance policy（閾値なし厳格運用）
- 絶対値閾値の達成ではなく、破綻を見逃さないことを合格条件にする
- 合格/不合格は以下で判定する:
  - 数値有限性（NaN/Inf禁止）
  - 診断列の継続性（欠損禁止）
  - 破綻前兆の検知（急激悪化、退化カウンタ増加、早期first-fail）

### 4. Verification protocol（必須検証）
- 設定テスト: modeと既存キーの整合
- 構造テスト: modeごとの構造差分
- 短時間統合テスト: phase別 short run のCSV検証
- 再現性確認: 複数seedで極端な早期破綻が常態化していないこと

### 5. Diagnostics compatibility policy（必須）
- 主要diagnostics列の無断削除・改名を禁止する
- 判定の source of truth は step_summary.csv に固定する
- 画像・動画の目視確認のみで合否を決めない

### 6. Runtime / output policy（必須）
- scripts と src の責務分離を維持する（アルゴリズム本体は src）
- outputs 配下の run.log / manifest.json 運用を崩さない
- 環境要因で未実行項目が出ても、差分は commit/push し、未実行理由を記録する

### 7. Mid-term hardening target（追補）
- 今回は short run を必須とする
- ただし中期的には multi-step（N=1000〜2000 step）検証を hard化対象として扱う

## テスト設計方針（甘くしない）

### Invariant tests
- NaN/Inf を1回でも検出したら失敗
- 必須diagnostics列が欠損したら失敗
- modeに応じた構造が一致しなければ失敗

### Consistency tests
- 同一configの複数seed実行で、first-fail step が極端に前倒しになるケースを失敗扱い
- mode間で想定される列セットが変化・消失した場合は失敗

### Trend-based failure tests
- 単一固定閾値ではなく、以下を失敗条件とする:
  - 局所誤差の急峻な増加
  - motor diagnostics の退化指標の連続悪化
  - local/global診断値の非有限化または欠損化

## 実装方針

### 力学モード
- dynamics_mode を導入し、構造切替のみを担わせる
- motor on/off は既存の motor.torque_Nm で制御する

### 既存phaseとの関係

- ただし内部実装は dynamics_mode ベースに寄せる

### source of truth
- 判定は step_summary.csv を最優先とする
- 画像・動画は補助情報扱い

### 移行期ルール
- dynamics_mode 導入後も phase0a/0b/1/2/3 の互換運用を維持する
- mode と n_flagella/stub_mode が矛盾する場合の優先順位・正規化方針を明示し、テストで担保する
