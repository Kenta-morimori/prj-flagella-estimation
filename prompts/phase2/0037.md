## 背景
Issue #37 の目的は、**トルクあり条件での安定回転**を、切り分け可能な段階構成で実現することである。

現状は以下の整理で考える。

- body-only の静的安定は概ね確認済み
- `minimal_basal_stub` の静的安定（motor off）は概ね確認済み
- full flagella の静的形状保持は未解決部分が残る
- motor ON 安定化は、現状では body / basal / full chain のどこが主因かが混ざりやすい
- そのため、`body only -> body + hook/static -> minimal_basal_stub + motor -> full_flagella + motor` の順で段階的に切り分ける

今回は**本質部分の切り分けとテスト/診断基盤の整備**を優先し、幾何の別論点（例: 7.5b 整合）は本PRに含めない。

## 目的
今回の主目的は、**1本べん毛条件で、トルクあり安定回転へ向かう段階的ゲートを定義し、実装・テスト・診断を整理すること**である。

最終 DoD は以下とする。

- `n_flagella=1`
- `stub_mode=full_flagella`
- `motor.torque_Nm != 0`
- run fixed（switching off）
- 診断可能な状態で短時間安定回転へ進める土台を作ること

ただし今回 PR では、最終 2 s hard gate をいきなり固定せず、**段階ごとのゲートと短時間レポートの枠組み整備**を優先する。

## スコープ
今回の実装スコープは以下とする。

1. フェーズ分割の明確化
2. 各フェーズの canonical config / 実行条件の整理
3. フェーズごとの hard test と report-only 実行の切り分け
4. actual motor torque をどの段階で入れるかの明確化
5. diagnostics / docs 更新ルールの追加

## フェーズ定義
本タスクでは以下のフェーズ定義を採用する。

### Phase0a: body-only static
目的:
- body 単体の有限性・静的安定を確認する

条件:
- `n_flagella=0`
- `motor.torque_Nm=0`
- body-only
- Brownian off
- switching off

期待:
- finite 完走
- NaN/Inf なし
- body diagnostics が出力される

### Phase0b: body + hook + minimal stub static
目的:
- hook と短い 3-bead flagellar stub（`minimal_basal_stub` モード）を有効にしても、motor off 条件で破綻しないことを確認する

条件:
- `n_flagella=1`
- `stub_mode=minimal_basal_stub`
- `motor.torque_Nm=0`
- Brownian off
- switching off

期待:
- finite 完走
- hook/basal 近傍の diagnostics が出力される
- `local_attach_first_rel_err` と `local_first_second_rel_err` が静的として妥当な範囲に収まる

### Phase1: body-only surrogate torque
目的:
- actual motor を入れる前に、body が純回転入力に耐えるかを切り分ける

条件:
- `n_flagella=0`
- `motor.torque_Nm=0`
- `body_equiv_load.enabled=true`
- `body_equiv_load.mode=pure_couple`
- Brownian off
- switching off

補足:
- ここでいう surrogate torque は、actual motor ではなく、body の回転応答のみを見るための代理トルク入力である
- `body_equiv_load.pure_couple` はその具体実装である

期待:
- finite 完走
- body-only 条件で pure couple による回転入力が診断可能である
- body diagnostics から回転入力時の body 安定性を確認できる

### Phase2: minimal_basal_stub + actual motor
目的:
- actual motor torque を最初に投入する段階
- basal/hook 近傍の破綻が主因かどうかを切り分ける

条件:
- `n_flagella=1`
- `stub_mode=minimal_basal_stub`
- `motor.torque_Nm != 0`
- Brownian off
- switching off

補足:
- `minimal_basal_stub` は body + hook のみではなく、**短い 3-bead flagellar stub を含む最小 basal 構成**である
- ここでの主観測対象は attach-first-second 近傍の局所崩れである

期待:
- finite 完走または少なくとも first-fail が診断可能
- `motor_*` と `local_*` diagnostics から、motor split / basal stretch / bend / torsion のどれが先に悪化するかを把握できる

### Phase3: full_flagella + actual motor
目的:
- full chain へ拡張したときの安定性を確認する
- full chain / torsion / chain maintenance 側の問題を切り分ける

条件:
- `n_flagella=1`
- `stub_mode=full_flagella`
- `motor.torque_Nm != 0`
- Brownian off
- switching off

期待:
- finite 完走または first-fail が診断可能
- Phase2 と比較して、full chain 固有の崩れがどこから立つか説明できる

## actual motor torque の扱い
今回の方針として、**actual motor torque は Phase1 には入れない**こと。

理由:
- Phase1 は body 側の純回転入力耐性を切り分ける段階であり、actual motor force split を混ぜると原因が混ざるため
- actual motor torque の初投入は **Phase2 (`minimal_basal_stub`)** とする
- full flagella + motor は **Phase3** で評価する

## テスト方針
基本方針は **TDD** とする。  
ただし、意味の薄い甘いテストは禁止する。

### hard test にするもの
以下は pytest で hard gate にする。

1. canonical config が各 phase で構築できること
2. body-only / motor off の finite 完走
3. minimal_basal_stub / motor off の finite 完走
4. body-only + pure_couple の finite 完走と diagnostics 出力
5. minimal_basal_stub + motor on の短時間実行で diagnostics が出力されること
6. full_flagella + motor on の短時間実行で diagnostics が出力されること
7. diagnostics docs 更新ルールに反しないこと（少なくとも docs ファイルの存在と更新前提を prompt に明記）

### hard test にしないもの
以下は pytest の pass/fail にせず、Codex が短いレポートで記録する。

- `0.01 s` の exploratory 実行
- `0.1 s` の first-fail 観測
- `2.0 s` の長時間 exploratory 実行
- 厳密な安定閾値の探索
- どの diagnostics が最初に崩れるかの定性的比較

### report-only 実行
Codex は少なくとも以下を report-only で実行し、結果を短くまとめること。

- Phase0a: `0.01 s`
- Phase0b: `0.01 s`
- Phase1: `0.01 s`
- Phase2: `0.01 s`
- Phase3: `0.01 s`

可能であれば `0.1 s` も比較すること。

レポートには最低限以下を含めること。

- phase 名
- config の主要差分
- finite / non-finite
- first-fail があればその step
- 主要 diagnostics の last step 値
- 読み取れる主因候補（body / basal / full chain）
