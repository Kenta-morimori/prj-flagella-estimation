## 本ドキュメントの目的
本タスクの第一目的は、力学演算を以下の3モードで明示的に扱えるようにし、
演算中に菌体・べん毛が弾ける挙動を許容しない検証枠組みを整えることである。

- body_only
- body_hook
- body_hook_flagella

加えて、論文モデルに近づけるために以下を進める。

- body / hook / flagella に対する projection を完全廃止する
- motor の数値安定化用 preload は段階的に削減し、最終的にポテンシャル主導の更新へ寄せる
- motor の各種 *_scale は当面は設定可能な補助項として残し、body + hook の sweep から順に弱める
- 最終的にはこれらの scale を全廃できるのが理想だが、現実的には可観測性と安定性を保ちながら段階的に外す

本フェーズでは、固定の絶対閾値を先に決めることはしない。
代わりに、破綻を取りこぼさない厳格テスト（不変条件・一貫性・悪化検知）を優先する。

## DoD

### PhaseA DoD（必須, torque=0, 図なし）
1. 1本べん毛・トルクなしで短時間安定化できること
- 想定構成: n_flagella=1, stub_mode=minimal_basal_stub, motor.torque_Nm=0
- 評価窓: short run（最低 0.01s）に加え、0.05s でも確認すること
- 全stepで以下を満たす:
  - pos_all_finite=True
  - any_nan=False
  - any_inf=False
  - shape_pass_nonbody=True
  - first_fail_category_nonbody=none
- body diagnostics が欠損せず、body shape 指標が継続出力されること

2. full_flagella + motor off の静止実行を厳格にクリアできること
- 想定構成: n_flagella=1, stub_mode=full_flagella, motor.torque_Nm=0
- 判定は `pos_all_finite=True` だけでは不十分で、形状崩壊を取りこぼさないこと
- 必須条件:
  - `hook_count == 1` と `flag_intra_count == 10` が全 step で維持されること
  - `hook_len_mean_over_b`, `local_attach_first_rel_err`, `flag_bond_rel_err_max`, `local_first_torsion_err_deg` が late-time で runaway growth しないこと
- クリア条件は step_summary.csv の時系列で判定し、見た目の印象では決めないこと

3. 構造切替が正しく反映されること
- body_only / body_hook / body_hook_flagella の各モードで、構築される構造（bead/spring/triplet/quad）が意図どおりであること

4. 可視化運用
- PhaseA では図の作成を実施しない（CSV 判定のみ）

### PhaseB1 DoD（必須, 観測義務）
1. トルクあり短時間ランの first-fail 判定を明文化すること
- 判定は `step_summary.csv` と body diagnostics から行い、以下のいずれかを first-fail と定義する:
  - `pos_all_finite=False` または `any_nan=True` または `any_inf=True`
  - 必須diagnostics列の欠損
  - 必須diagnostics列の非数値化/非有限化
  - motor 退化カウンタの増加（`motor_degenerate_axis_count`, `motor_split_rank_deficient_count`, `motor_bond_length_clipped_count`）
  - shape gate 不通過（non-body または body）

2. B1 の目的は「安定合格」ではなく、失敗分類の再現性固定である
- 想定構成:
  - PhaseB1-minimal: n_flagella=1, stub_mode=minimal_basal_stub, motor.torque_Nm!=0
  - PhaseB1-full: n_flagella=1, stub_mode=full_flagella, motor.torque_Nm!=0
- 失敗時に first-fail step と reason が特定できること

3. 可視化運用
- 図作成は PhaseB1 以降で開始する
- 初回は torque×scale の2軸可視化を `local_hook_scale` で実施する
- 可視化は body / hook / flagella の pass/fail heatmap とする
- body と hook は常時出力する
- flagella は flag 系の first-fail がある場合にのみ出力する
- 付随的な category heatmap は診断用として残してよい

### Update (2026-04-14 / visualization separation)
- Component pass/fail 判定を「first-fail category 優先度」から分離する
- 新しい判定軸:
  - **独立判定軸**: 各コンポーネント（body/hook/flagella）が「崩壊しているか」を直接判定
    - body: `body_shape_pass` 列を直接使用（spring/bend/centerline/area 閾値に基づく）
    - hook/flagella: `first_fail_category` に基づく（実装上の区別は運用効率のため）
  - **診断軸**: 整体的に「最初に誘発された失敗は何か」を `first_fail_category` で記録
- 可視化出力の変更:
  - メイン出力: 1×3 subplot (body/hook/flagella pass/fail) を新規出力
    - ファイル名: `*_combined_pass_fail_heatmap.png`
    - 色付け: 赤=#c53030 (fail), 緑=#2f855a (pass)
  - 診断出力: 既存の個別 heatmap + category heatmap を継続出力
- 効果:
  - 「body は正常に分類されるがメトリクスが悪化している」という矛盾を可視化時に検出可能に
  - body のメトリクス悪化を主観ではなくデータ駆動で判定できる

### PhaseB2 DoD（必須, hard gate）
1. B1 で確立した判定を pytest の fail 条件に昇格する
- `finite_pass=True` でも shape gate fail なら不合格とする

2. 既知失敗ケースを管理し、修正後に pass へ反転させる
- 既知失敗例: n_flagella=1, minimal_basal_stub, torque=4.0e-21, ramp=false, local_hook_scale=8.0, duration=0.05s

### PhaseC DoD（必須, sweep再開）
1. B2 を満たす基準トルクを再同定する
2. 基準トルク固定で scale sweep を再開する
- 順序: local_hook_scale → local_spring_scale → local_bend_scale/local_torsion_scale
3. 可視化は pass/fail と first_fail_category を主軸に比較する

## テスト設計方針（甘くしない）

### Invariant tests
- NaN/Inf を1回でも検出したら失敗
- 必須diagnostics列が欠損したら失敗
- modeに応じた構造が一致しなければ失敗

### Consistency tests
- 同一configの複数seed実行で、first-fail step が極端に前倒しになるケースを失敗扱い
- mode間で想定される列セットが変化・消失した場合は失敗

### Trend-based failure tests
- 単一固定閾値ではなく、以下を失敗条件とする:
  - 局所誤差の急峻な増加
  - motor diagnostics の退化指標の連続悪化
  - local/global診断値の非有限化または欠損化

## 実装方針

### 力学モード
- dynamics_mode を導入し、構造切替のみを担わせる
- motor on/off は既存の motor.torque_Nm で制御する

### 拘束・安定化方針
- projection 系は実装・参照・診断の残骸も含めて段階的に削除する
- motor の preload は一度に全削除せず、影響範囲を区切って段階的に外す
- 変更のたびに short run で step_summary.csv を確認し、first-fail の有無を判定する
- torque ramp は default off を維持し、必要時のみ短時間の補助的な確認用に使う
- weighted minimum-norm による motor 力分配は、当面は数値安定化の暫定策として残すが、
  short run で simpler coupling との差分を比較しつつ段階的に簡素化する
- flagella の attachment angle / self-rotation は diagnostics で固定的に追跡する
  - `flag_root_azimuth_deg`
  - `flag_phase_deg`
  - `flag_phase_rate_hz`
  - `flag_body_phase_diff_deg`
  - これらは `step_summary.csv` に必ず出力し、無断削除・改名を禁止する
- 実験は torque ramp なしを基本とし、`body + hook` から始めて hook だけを弱める観測 sweep を先行する
- 崩壊が見えた場合は、`local_attach_first_rel_err` / `hook_angle_err_*` / `flag_*` のどこが先行したかを分析する

### 既存phaseとの関係
- phase0a/0b/1/2/3 の名称は互換維持
- ただし内部実装は dynamics_mode ベースに寄せる

### source of truth
- 判定は step_summary.csv を最優先とする
- 画像・動画は補助情報扱い

### 移行期ルール
- dynamics_mode 導入後も phase0a/0b/1/2/3 の互換運用を維持する
- mode と n_flagella/stub_mode が矛盾する場合の優先順位・正規化方針を明示し、テストで担保する
