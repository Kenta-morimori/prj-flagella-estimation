## 目的
1. **spring自然長を「初期配置の距離」から決める**（body/flag/body-flag を例外なく統一）。
2. **フック長（body–flag基部距離）を 0.25b** として初期配置に反映する。
3. MVP制約として **n_prism=3 かつ 0<=n_flagella<=3** のみを許容し、それ以外はエラーにする。
4. 既存構造（ModelBuilder, spring_pairs, hook_triplets, logging など）を可能な限り維持する。

---

## 背景（現状の問題）
- flagella の隣接ボンド自然長が固定値（bond_L_over_b*b）で、helix幾何の初期隣接距離と一致しない場合、初手から大きな spring 力が出る → 発散。
- body–flag の基部が attach と同位置に置かれており、フック長 0.25b が初期位置に反映されていない。
- n_prism や n_flagella の自由度があるが、MVPでは構造を限定して検証したい。

## DoD（完了条件）
- `motor.torque_Nm=0` かつ `hook.enabled=false` でも、初期数ステップで「べん毛が飛散」しない（少なくとも mean_disp が極端に大きくならない）。
- `spring_rest_lengths_m` は **全 spring_pairs** について `L0 = ||x_j - x_i|| (t=0)` を採用している。
- フック長は初期配置で **||x_flag_base - x_attach|| = 0.25b** になる。
- MVP制約：
  - `n_prism != 3` → ValueError
  - `n_flagella < 0 or n_flagella > 3` → ValueError

## タスク（できるだけ最小差分で）

#### 方針
- 早い段階（ModelBuilder.build か params生成直後）で検査して例外にする。
- 例外メッセージはユーザが設定値を直せる形にする。

#### 実装案

  - `if cfg.body.n_prism != 3: raise ValueError("MVP: body.n_prism must be 3")`
  - `if not (0 <= cfg.flagella.n_flagella <= 3): raise ValueError("MVP: flagella.n_flagella must be in [0,3]")`

（CLI側で弾くより builder で弾く方が、後続処理を壊さない。）

---

### Commit 2: フック長 0.25b を初期配置に反映（最小破壊）
#### 目標
- flagella の基部点（flag bead #0）が attach と同一点にならないようにし、
  `||flag0 - attach|| = hook_length = 0.25*b` を満たす。

#### 変更方針（既存を壊さない）
- flagella座標生成の「平行移動」の部分で、原点オフセットを追加するだけに留める。
- hook方向は「その層の断面中心→attach 方向（外向き）」を使用する（安定で実装が軽い）。

#### 実装仕様
- `builder.py` で body の層 `center_layer = body_layers[len(body_layers)//2]` を用いる（既に中央層attachに変更済みならそのまま）。
- 各 attach_idx について：
  - `center = mean(body_um[center_layer], axis=0)` を層中心とする
  - `u = normalize(body_um[attach_idx] - center)`（外向き単位ベクトル）
  - `hook_offset_um = (0.25 * b_um) * u`
- flagella points を平行移動するときに
  - `flag_points = local_points + body_um[attach_idx] + hook_offset_um`
  とする
- これにより `flag_points[0]` が attach から 0.25b 離れる。

---

### Commit 3: spring自然長を「初期距離」へ統一（全 spring_pairs）
#### 方針
- 既存の spring_pairs 生成ロジックは残す。
- ただし `spring_rest_lengths_m` は、buildの最後に **一括で再計算して上書き**する。
  - bodyの ring/vertical で既に距離を入れている箇所もあるが、最終的に統一されるので問題ない。
- これにより flagella の固定自然長が原因のズレを確実に排除できる。

#### 実装仕様
- `builder.build()` で positions_m（最終座標）が確定し、spring_pairs が完成した直後に：
  - `spring_rest_lengths_m = [||pos[j]-pos[i]|| for (i,j) in spring_pairs]`
  を必ず実行する
- 以後 `compute_spring_forces()` はこの `spring_rest_lengths_m` を参照する前提。
  - すでに参照しているならそのまま
  - もし一様L0前提が残っているなら、配列参照に統一する

---

### Commit 4: body–flag（attach–flag0）を spring+hook として確実に保持
#### 方針
- 既に spring+hook が入っている場合は、そのまま維持。
- まだなら、以下を追加：
  - `spring_pairs.append((attach_idx, flag_idx0))`
  - `hook_triplets.append((attach_idx, flag_idx0, flag_idx1))`
- Commit 2 の hook_offset により、attach–flag0 の初期距離は 0.25b になる。
- Commit 3 により、この spring の自然長も初期距離（0.25b）になる。

## 参考：現状 n_prism≠3 / n_flagella>3 だとどうなるか（現状挙動）
- n_prism≠3：
  - body断面は n_prism点の多角形（円周上配置）になり、正三角形柱ではなくなる（ただしコード自体は動く）。
- n_flagella>3：
  - attachは層のビーズから「等間隔に近い」選び方になるが、密になり衝突や数値不安定が起きやすい。
  - 可視化の凡例・色・配列前提が崩れる可能性もある。
MVPではこれらを避けるため、今回の制約でエラーにする。
